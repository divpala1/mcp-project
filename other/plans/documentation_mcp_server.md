# Documentation MCP Server — Design Plan

**Status:** Proposal. Not yet built.
**Replaces:** the placeholder `mcp_server/` (RAG-over-arbitrary-content) and `notes_server/`. Those go away once this server lands.
**Audience for this doc:** the engineer building the server (you). It explains *what* to build, *why* each choice was made, and *what to leave for later*.

---

## 1. What this server is

A single-purpose MCP server whose job is: **let any agent answer end-user questions about the Daitics AI CDP application by retrieving the right pieces of its user-facing documentation.**

The docs themselves are markdown files generated by the `documentation_generation_prompt.md` workflow — one repo, one folder, written by humans and/or our docs-generator agent, committed to source control.

This server exposes those docs to LLM agents through a small, opinionated tool surface tuned for retrieval quality, not for raw flexibility.

### What this server is *not*

- Not a generic RAG-over-arbitrary-files engine. It assumes the docs follow the structure mandated by `documentation_generation_prompt.md` and exploits that structure.
- Not a multi-tenant content store. Documentation is global to the product; tenant scoping does not apply here. (See §8 — auth.)
- Not a doc *editor*. Docs are written in markdown, committed to git, and ingested at server startup. There is no `create_section` / `edit_section` MCP tool. If the docs change, the index reloads. Editing happens via PRs to the docs repo.
- Not a chatbot. It retrieves; the agent reasons.

---

## 2. Design north star

Three constraints, in priority order:

1. **The agent always finds the right section.** Retrieval quality is the entire game. A mediocre retriever turns a 70B-parameter model into a wrong-answer machine; a good retriever lets a 7B-parameter model answer correctly.
2. **Small models can use it as easily as large models.** The tool surface must be tiny, the tool descriptions must be self-documenting, and tool outputs must be structured (predictable JSON, consistent field names) so a 7B–13B model doesn't have to reason about response *format* on top of *content*.
3. **Boring to operate.** One process, one Qdrant collection, one config file. No background workers, no eventually-consistent indexes, no compaction jobs.

Every choice in this doc traces back to one of those three.

## 3. The tool surface

The server exposes **five MCP tools**, all prefixed `docs_`. All are read-only.

| Tool | Purpose | Why it exists |
|---|---|---|
| `docs_search(query, top_k=5)` | Semantic search over all sections; returns top-k matches with a snippet, breadcrumb, score, and stable section ID. | The workhorse. ~80% of agent calls will be this one. |
| `docs_browse(category=None)` | Lists section titles + IDs, optionally filtered by category (`getting-started`, `features`, `how-to`, `troubleshooting`, `glossary`). Pure metadata, no content. | For "what can the app do?" / "show me how-to guides" queries — pagination not required at expected corpus size, but the response is bounded by category, which gives natural granularity. |
| `docs_get(section_id)` | Returns the full markdown of one section, plus its metadata. | When `docs_search` returns a snippet that needs full context (steps that got truncated, surrounding bullets, code blocks). |
| `docs_glossary(term)` | Exact + alias-aware lookup of a single glossary term. | Glossary terms are short, frequently asked about by exact name, and embedding-search underperforms on rare proper nouns. A dedicated path is faster, cheaper, and more accurate. |
| `docs_overview()` | Returns the doc set's table of contents + 1-paragraph product description. Small, cheap, idempotent. | Lets the agent build a mental map on first contact ("what kind of app is this?") without burning a search call on a generic query. |

### Why exactly five — not three, not ten

- **Three would be too few.** A search-only surface forces the LLM to embed every navigation question ("list features", "what's in the glossary?") as a search query, which wastes embedding compute and produces noisy results. `docs_browse` and `docs_glossary` are exact-match paths that beat semantic search for those cases.
- **Ten would be too many.** Per CLAUDE.md C2, LLMs degrade past ~10 tools. Adding tools like `docs_related(section_id)`, `docs_recent_changes()`, `docs_search_by_category(...)` is tempting but each one is a tax on the LLM's tool-selection step. Anything that *can* be expressed via `search` + `get` should not get its own tool.
- **Why no `ingest` tool.** Docs are committed to git and read by the server. A user-facing `ingest` tool would be a design hole — anyone with a bearer token could pollute the official docs. Indexing is a server-side concern triggered at startup or via an admin reload (§5).

### Output shapes — the same fields everywhere

A consistent schema across tools matters more than richness. The LLM learns the shape once and reuses it.

```jsonc
// docs_search → { results, count }
{
  "results": [
    {
      "section_id": "how-to/export-data#csv-export",
      "title": "How to export your data as CSV",
      "category": "how-to",
      "breadcrumb": "How-To › Export › CSV export",
      "snippet": "To export data from the dashboard as CSV…",   // ≤ 500 chars
      "snippet_truncated": true,
      "score": 0.83
    }
  ],
  "count": 1
}

// docs_browse → { sections, count }
{
  "sections": [
    { "section_id": "...", "title": "...", "category": "...", "summary": "..." }
  ],
  "count": 12
}

// docs_get → full section
{
  "section_id": "how-to/export-data#csv-export",
  "title": "How to export your data as CSV",
  "category": "how-to",
  "breadcrumb": "How-To › Export › CSV export",
  "content": "<full markdown>",                                // ≤ 8 KB; truncated flag if over
  "tags": ["export", "csv", "dashboard"],
  "aliases": ["download data", "save spreadsheet"],
  "last_updated": "2026-04-12",
  "truncated": false
}

// docs_glossary → single entry or null
{
  "term": "Audience",
  "definition": "...",
  "aliases": ["segment", "user group"],
  "section_id": "glossary#audience"
}

// docs_overview → small, cheap
{
  "product_name": "Daitics AI CDP",
  "tagline": "...",
  "table_of_contents": [
    { "category": "getting-started", "section_count": 3 },
    { "category": "features",        "section_count": 14 },
    { "category": "how-to",          "section_count": 22 },
    { "category": "troubleshooting", "section_count": 1  },
    { "category": "glossary",        "section_count": 31 }
  ],
  "version": "2026-05-01"
}
```

**Why these fields.**
- `section_id` is the addressing primitive — stable across reindexes, format `{category}/{filename}#{slug}`. The agent uses this with `docs_get`. Stable IDs are the single biggest determinant of whether `search → get` chains work cleanly.
- `breadcrumb` is human- and LLM-readable in one shot. No looking up parent files.
- `score` is exposed so the agent can decide whether to trust a result or hedge ("I found this but the relevance score is low").
- `snippet_truncated` / `truncated` flags are the C5 pattern — the agent *sees* the truncation and can call `docs_get` to fix it.

## 4. The unit of retrieval: a "section"

A **section** is one H2-heading block in a markdown file plus everything under it up to the next H2 (or end of file).

```markdown
# How to export your data       ← H1: page title (one per file)

## CSV export                   ← H2: section. Becomes a retrievable chunk.
### What this does              ← H3 stays inside the parent section
...
## JSON export                  ← H2: a different section
...
```

### Why H2 and not paragraphs / fixed tokens / files

- **Fixed-size chunking** loses semantic boundaries. A "Steps" list split mid-list is useless to the agent.
- **Whole files** are too coarse. A page about "Export" might cover CSV, JSON, and XML — the agent should be able to retrieve only the CSV part.
- **H2 sections** are exactly the boundary the documentation-generation prompt already enforces ("one topic per section", "front-load the answer"). Using the structure the writer already built is cheaper, more accurate, and self-correcting — the writer's edits to a section directly improve retrieval.

### Section identity

```
section_id = "<category>/<file_basename>#<heading_slug>"

Examples:
  how-to/reset-password#default-flow
  features/audience-builder#what-it-does
  glossary#audience
  troubleshooting#export-fails
```

- Filenames and heading slugs are the source of truth.
- IDs are produced deterministically from the markdown — no random UUIDs, no DB-assigned numbers. Same docs in, same IDs out, every time. This is what makes `docs_get(section_id)` safe for an agent to pass section IDs around in conversation without them going stale across reindexes.
- IDs are URL-safe and copyable, so they can also be shown to users ("see section `how-to/reset-password#default-flow`").

### What goes into the embedding vs. the payload

| Field | Embedded? | Payload? | Why |
|---|---|---|---|
| Section title | Yes | Yes | Most search-able piece. Front-loaded by design. |
| Breadcrumb (H1 + H2) | Yes | Yes | Adds parent context — "Export › CSV" beats "CSV export" alone. |
| Section body | Yes | Yes | The content. |
| `aliases` (frontmatter) | Yes (concatenated) | Yes | Lets a search for "log in" hit a section titled "Sign in". |
| `tags` | No | Yes | For future filter UIs; not discriminative enough to embed. |
| `category` | No | Yes (indexed) | Used as a Qdrant filter for `docs_browse`. |
| `last_updated`, `version` | No | Yes | Surfaces in tool output; useful for the agent to qualify staleness. |

Embedding the breadcrumb + aliases alongside the body gives the retriever crucial extra surface area at zero cost — the section text already implicitly contains the title via the writing-style rule "repeat the subject inside the section", but explicit > implicit.

## 5. Indexing pipeline

```
docs/                                ← source of truth, in git
  index.md
  getting-started.md
  features/*.md
  how-to/*.md
  troubleshooting.md
  glossary.md
        │
        ▼
  parser.py                          ← reads file, extracts YAML frontmatter,
        │                             splits by H2, computes section_id +
        │                             content_hash
        ▼
  indexer.py                         ← embeds, upserts to Qdrant. Skips
        │                             sections whose content_hash matches
        ▼                             the stored payload (idempotent reload)
  Qdrant collection `documentation`
```

### Triggering indexing

- **At server startup (always).** The FastAPI lifespan event walks `DOCS_PATH`, builds the index, then yields. If the docs directory is missing or empty, fail fast — that is a deployment misconfiguration, not a runtime condition to recover from.
- **On demand via `POST /api/admin/reload`.** Bearer-token-protected REST endpoint (no MCP tool — see §3). Lets us push doc changes without restarting the process. Reload is incremental: sections whose `content_hash` already matches the stored payload are skipped, so a 1-section edit doesn't re-embed the entire corpus.
- **(Future) File-watcher.** A `watchdog`-based watcher could trigger reload on filesystem changes. Not in scope for v1; flag with `# TODO(future): file watcher`.

### Why content-hash incremental reindex

Embedding 200 sections takes a few seconds, not minutes — so full reindex is technically fine. But:
- It's the only way to keep `last_updated` honest at the section level.
- It surfaces "ghost" sections — IDs in Qdrant that no longer exist in `docs/` are deleted on reload, preventing the index from drifting away from the source.
- It is genuinely simple: hash the canonicalised markdown of a section, compare, skip-or-embed.

### Why one Qdrant collection per server, not per category

Categories (`features`, `how-to`, etc.) are small and queries often span them ("how do I do X?" might land in either `features/X.md` or `how-to/X.md`). A single collection with a `category` payload index gives O(1) filtered browse and zero overhead for unfiltered search. Per-category collections would force two-stage retrieval and add operational fan-out for no win.

## 6. Search & ranking

### v1 — semantic primary, exact-match for glossary

- **`docs_search`:** sentence-transformer embedding (`all-MiniLM-L6-v2`, same model the project already uses) → Qdrant cosine similarity → top-k.
- **`docs_glossary`:** plain-Python dictionary lookup keyed on `term.lower()` and every entry in `aliases.lower()`. No embedding required.
- **`docs_browse`:** Qdrant filter on `category`, no vector query. Returns titles + IDs + 1-line summaries (from frontmatter).

### Why semantic-only for v1, not hybrid

Hybrid search (semantic + BM25, fused via RRF) is genuinely better for technical documentation — users mix natural-language phrasing with exact product names ("export the SCAR widget"). However:

- The first-class glossary tool already absorbs a large fraction of the "exact term" queries.
- BM25 adds: a second index to maintain, a fusion step, more code paths to test, and a small extra dependency (`rank_bm25` or Qdrant's named-sparse-vectors).
- The win is real but bounded — typically +5 to +10 points on retrieval@5 for product docs in published evaluations.
- **Plan, but do not ship.** Hybrid is the v2 upgrade and §11 names exactly where it slots in.

This is the right call for the project's "simple but extensible" goal. The shape we ship is the shape that future-hybrid drops into without a rewrite.

### Re-ranking, query expansion — explicit non-goals for v1

Both materially improve retrieval (cross-encoder rerank, LLM-based query rewriting). Both add latency, cost, and complexity. Both are well-known seams that go in front of the existing retriever. They are flagged in §11 with implementation hints. Not in v1.

### Score thresholds

`docs_search` does **not** apply a score cutoff. The agent sees the score and decides; thresholding inside the tool would amount to silently lying about empty results when a low-confidence match exists. The system prompt should be the place that tells the agent how to interpret low scores.

## 7. Document model — concrete schema

### Required YAML frontmatter (per file)

The documentation-generation prompt does not currently mandate frontmatter. **It should.** §10 below proposes the exact addition.

```yaml
---
id: how-to/reset-password           # path-form id, no slashes in heading slugs
title: How to reset your password
category: how-to                    # one of: getting-started | features | how-to | troubleshooting | glossary | overview
summary: Reset your password from the sign-in screen via the "Forgot password" link.
tags: [account, security, password]
aliases:                            # synonyms users might type
  - lost password
  - can't sign in
  - forgot password
last_updated: 2026-05-01
version: 1
---
```

### Section-level metadata (in-line)

Per section under each H2, an optional `<!-- aliases: x, y, z -->` HTML comment lets a writer attach aliases to a specific section without bloating frontmatter. Parser reads these. Optional and nice-to-have; not required for v1 to work.

### Glossary file

`glossary.md` is special-cased: every H2 inside it is a term, and the parser stores entries in a flat `{term, definition, aliases, section_id}` dict in addition to the normal section index. This is what `docs_glossary` queries.

## 8. Auth model

**Bearer token, same pattern as the rest of the project.** Reasons:

- Consistency with `mcp_server` / `notes_server` reduces cognitive load.
- The agent already passes `Authorization: Bearer <token>` on every MCP call — zero new wiring.
- Even though docs are global, *who is reading* matters for usage analytics and future per-tier doc gating.

**No tenant filtering.** Documentation is global to the app. The token is verified (any valid token works) but no `org_id` filter is applied to retrieval. A `# NOTE` in the source notes the natural extension point: per-org doc overlays (e.g. enterprise-tier-only sections) would slot in as a Qdrant payload filter on a `tier` field, mirroring the C1 pattern from the placeholder server.

**`/api/health` and `/api/admin/reload`** behave consistently with the existing pattern: health is open, reload requires a bearer.

## 9. Configuration

Add to `.env`:

```env
# Documentation MCP server
DOCS_PATH=./docs                                # filesystem path to the markdown corpus
DOCS_QDRANT_COLLECTION=documentation            # separate from the placeholder RAG collection
DOCS_RELOAD_TOKEN_SCOPE=admin                   # which scope/role the bearer must carry to call /admin/reload
```

Reuse from existing config:
- `EMBEDDING_MODEL` — same model, same dimension, same singleton loader.
- `QDRANT_URL` — same Qdrant instance, different collection.
- `AUTH_TOKENS_JSON` — same token map; eventually grows a `scope` field per token (`{user_id, org_id, scopes: ["read", "admin"]}`) so reload can be locked to admins. For v1, treat any valid token as authorized for reload and `# TODO(future)` the scope check.

Why a separate Qdrant collection: keeps the documentation index isolated from any other RAG content the project might end up indexing. Different lifecycle (rebuilt from `docs/`, not user-uploaded), different schema (richer payload), different access pattern (read-only). One collection per logical domain is the right granularity.

## 10. Suggested updates to `documentation_generation_prompt.md`

The current prompt is excellent for human readability. It needs three small additions to be **retrieval-grade**.

### 10a. Mandate YAML frontmatter on every file

Add to the prompt's "File structure" step:

> Every markdown file MUST begin with YAML frontmatter:
> ```yaml
> ---
> id: <category>/<file-basename>
> title: <human title — same wording as the H1>
> category: <one of: getting-started | features | how-to | troubleshooting | glossary | overview>
> summary: <≤ 30 words. The pitch. Used by docs_browse and docs_overview.>
> tags: [<3–6 user-language keywords>]
> aliases: [<other phrasings of the title users would search for>]
> last_updated: YYYY-MM-DD
> version: <int, bumped on substantive changes>
> ---
> ```
> The MCP server uses these fields directly. `aliases` in particular is the writer's lever for catching alternate phrasings — pour real effort into it.

### 10b. Add an embedding-friendly first-sentence rule

Add to the "Writing style (non-negotiable)" list:

> **First sentence of each section must work as a standalone answer.** If the first 200 characters were retrieved with no other context — heading, surrounding text, page title — they should already start to answer the question the heading implies. The MCP server retrieves at the section level; the first sentence is what an LLM grades on.

This is a tightening of the existing rule "Front-load the answer", not a new constraint.

### 10c. Glossary entries get explicit synonym lists

Add to the prompt's section on `glossary.md`:

> Each H2 in `glossary.md` is one term. Under the term, write a 1–3 sentence plain-language definition. Then, on a single line at the end of the entry, add an HTML comment listing synonyms:
> ```
> <!-- aliases: segment, user group, cohort -->
> ```
> The MCP server's glossary tool uses this to resolve user phrasings to the canonical term.

### 10d. Stable section slugs

Add to the file template:

> Under each H2 heading, on the same line, append `{#kebab-slug}` to pin the section's slug:
> ```markdown
> ## CSV export {#csv-export}
> ```
> Without this, the slug is auto-generated from the heading text and would change if the heading is reworded — breaking any agent or human reference to the section ID. Pin the slug once and treat it as part of the URL.

### What stays as-is

The discovery/inventory step, the "humans + MCP both matter" framing, the writing-style banlist, and the quality-bar checklist are all pulling their weight already. Don't touch them.

## 11. Future-proofing — explicit extension seams

Each item below is a **named code location**, not a vibes-y promise.

| Future capability | Where it slots in | Approximate cost |
|---|---|---|
| **Hybrid search (BM25 + semantic, RRF fused)** | New `core/retriever.py` function. `docs_search` calls a `Retriever` interface; `SemanticRetriever` is v1, `HybridRetriever` is the drop-in. Qdrant supports sparse vectors natively, so no extra infra. | ~50 lines; one new dep (`rank_bm25`) optional if Qdrant sparse vectors are used instead. |
| **Cross-encoder re-ranking of top-N** | Wrap `docs_search`'s post-retrieval list with a `Reranker.rerank(query, candidates, top_k)` call. `NoopReranker` v1; `BgeReranker` (or any cross-encoder) v2. | ~30 lines + a model download. Adds 100–300 ms of latency; switchable by env var. |
| **LLM query rewriting** | Pre-retrieval hook in `docs_search` that calls a small LLM to expand the query into 2–3 paraphrases, retrieves on each, merges. Lives in `core/query_rewriter.py`. | ~40 lines; opt-in via `DOCS_QUERY_REWRITE=true`. |
| **File watcher for live reload** | New `core/watcher.py` using `watchdog`. Calls the same `indexer.reload()` the admin endpoint already exposes. | ~25 lines. |
| **Per-tier / per-org doc overlays** | Add `tier` to section payload + add to `_filter()` helper. Same C1 pattern as the placeholder server. | ~15 lines once the tier field exists in the frontmatter schema. |
| **Multi-language docs** | `docs/<lang>/...` plus a `language` payload field; resolve from request header or token claim. | Larger; out of v1 scope. |
| **Version-pinned docs (per app release)** | Add `app_version` to frontmatter and payload; `docs_search` accepts an optional `app_version` filter. | ~20 lines; depends on release pipeline. |

## 12. File layout

```
docs_server/                         ← new package; replaces mcp_server/ & notes_server/
├── server.py                        FastAPI + FastMCP entrypoint, lifespan = build_index()
├── auth.py                          bearer middleware + ContextVar (lift from mcp_server/)
├── core/
│   ├── config.py                    pydantic-settings
│   ├── parser.py                    markdown → list[Section]; reads frontmatter,
│   │                                splits by H2, computes content_hash, slugifies headings
│   ├── indexer.py                   walk DOCS_PATH, parse, embed, upsert; incremental reload
│   ├── retriever.py                 SemanticRetriever (v1) + interface for HybridRetriever (v2)
│   ├── glossary.py                  in-process dict; rebuilt on reload
│   ├── embeddings.py                singleton sentence-transformer (lift from mcp_server/)
│   └── store.py                     Qdrant client + filter helpers (org-scoping seam)
├── docs/                            ← actual markdown corpus. NOT in this folder forever:
│                                      this is just the dev convenience location. Production
│                                      reads DOCS_PATH from env.
└── WALKTHROUGH.md                   the same kind of file-by-file walkthrough mcp_server/ has
```

### Why a fresh package, not a refactor of `mcp_server/`

`mcp_server/` is a different domain (generic RAG, ingest API, multi-tenant by org). Mutating it into the docs server would lose the teaching value of comparing the two designs side by side. Once `docs_server/` is stable, `mcp_server/` and `notes_server/` get deleted in a single dedicated PR — no in-place mutation.

## 13. Migration plan (concrete steps)

**Phase 0 — alignment (this doc).** Done when this plan is reviewed and any objections are folded back in.

**Phase 1 — server skeleton.**
- Create `docs_server/` with `server.py`, `auth.py`, `core/config.py`, `core/embeddings.py` (lifted), `core/store.py` (Qdrant client + collection bootstrap).
- Wire `/api/health`, an empty FastMCP app, and the lifespan hook (no-op for now).
- Smoke test: `curl /api/health` and `mcp/sse` open + handshake.

**Phase 2 — parser + indexer.**
- Implement `parser.py` against a 5-file fixture corpus (one per category).
- Implement `indexer.py` with content-hash skip logic.
- Unit-test parser edge cases (no frontmatter → fail loudly; missing required fields → fail with field name; H1 without H2 → emits one section per file).

**Phase 3 — tools.**
- Implement `docs_search`, `docs_get`, `docs_browse`, `docs_glossary`, `docs_overview` in `server.py`.
- Each tool: ≤20 lines, delegates to one `core/` function.
- End-to-end smoke test against the fixture corpus.

**Phase 4 — admin reload + retriever interface.**
- `POST /api/admin/reload` endpoint.
- Refactor retrieval into a `Retriever` interface with `SemanticRetriever` as the implementation. (Sets up Phase 5b.)

**Phase 5a — agent system prompt update.**
- The agent currently has `docs_*` tools from the placeholder. Tool *names* collide with the new server's tools, but the *meanings* differ. Resolve by:
  - Wire only `docs_server` and remove the placeholders' MCP entries.
  - Update `agent/prompts/system.md` with a short blurb on how to use the docs tools (especially: prefer `docs_glossary` for proper-noun lookups).

**Phase 5b — drop the placeholders.**
- Delete `mcp_server/` and `notes_server/` once the docs server is verified end-to-end and the agent runs cleanly against it for a week of dev use.
- Update `README.md`, `CLAUDE.md`, `pyproject.toml`, `docker-compose.yml` (no Qdrant change needed — same instance, different collection).

**Phase 6 (optional, when needed) — hybrid search + reranker.** Per §11.

## 14. Testing strategy

Three layers, smallest first:

1. **Unit (fast).** Parser correctness on fixture markdown (frontmatter parsing, H2 splitting, slug pinning, content-hash stability). No Qdrant, no embeddings. Run on every commit.
2. **Component (medium).** `indexer.py` against a real Qdrant (already in `docker-compose`) — verify upsert, idempotent reload, ghost-section deletion.
3. **End-to-end retrieval quality (slow, manual at first).** A small evaluation set of 20–30 (question → expected_section_id) pairs. Run `docs_search` against each, measure recall@5 and MRR. This is the number that matters — it's how we'll know whether hybrid search and reranking actually help when Phase 6 happens.

Eval set lives in `docs_server/evals/` as a CSV. Run via a one-shot script (`python -m docs_server.evals.run`) — no test framework yet. When evals matter enough to be in CI, they get promoted; until then, this is the right amount of ceremony.

## 15. Things deliberately NOT in this plan (and why)

- **A web UI / admin console.** The MCP surface + `/api/admin/reload` is enough. A UI is a separate product.
- **Streaming tool responses.** Tool outputs here are small and bounded. Streaming inside a tool call is over-engineering; the agent's *output* still streams normally.
- **Document edit tools.** Writing happens in git, not over MCP. (See §1.)
- **Per-tool rate limiting.** Bearer-token rate limiting belongs at the gateway/proxy layer if it's needed at all. Don't bake it into the tool layer.
- **Versioned section IDs (`?v=42`).** Sections are mutable in place. Stable IDs + `last_updated` are enough; an agent that needs frozen content can fetch `docs_get` and snapshot the response.
- **Multi-language support.** Real concern, real complexity. Marked as a §11 future seam; not v1.

## 16. Open questions for review

These are the calls I made under uncertainty. Worth a 5-minute discussion before Phase 1.

1. **Where does the `docs/` corpus actually live in production?** Same repo as the agent? Separate repo, pulled at startup? Object storage? The plan assumes "filesystem path, configured via env" — that's the lowest-friction thing that works. Anything more elaborate is out of v1.
2. **Who owns reloads?** A human running `curl /api/admin/reload` after a docs PR merges is fine for now. Eventually this should be a CI step in the docs repo. Worth deciding before the docs-generation devs ship their first PR.
3. **Do we want per-tier docs from day one?** If the product has paid tiers with feature-gated docs, the cost of adding that *now* is small (one frontmatter field + one filter). Deferring it costs more if it's needed within a quarter. I've assumed not-yet, but happy to flip.
4. **Tool naming: `docs_*` is fine but slightly generic.** Alternatives: `helpdocs_*`, `appdocs_*`, `userdocs_*`. `docs_*` wins on brevity but loses ambiguously when the agent eventually has multiple "docs"-like sources (LangChain Docs MCP, GitHub repo READMEs, etc). Worth a 30-second naming call before Phase 1.
